## 2.5 JS transversales

### `nerisoft_navbar.js`

Responsable de la interaccion global del navbar y el mega menu. Su alcance debe mantenerse transversal: apertura/cierre, comportamiento de grupos y compatibilidad con el layout base. No debe contener reglas de negocio de Gestion o Contabilidad.

### `nerisoft_alerts.js`

Responsable del comportamiento de alertas globales. Se vincula al contrato UI de mensajes tipo modal/alert superior con cierre visual. No debe decidir mensajes funcionales; solo presentarlos.

### `nerisoft_select.js`

Motor global de pseudo-selects. Convierte selects marcados con `data-ns-select` en controles visuales uniformes. Modos vigentes:

- `data-ns-select="normal"`: select visual sin busqueda.
- `data-ns-select="search"`: select visual con input de busqueda.

Contrato vigente:

- El `<select>` nativo sigue siendo la fuente de verdad del valor enviado por formulario.
- El componente visual debe sincronizarse con `change`, `disabled`, opcion seleccionada y opciones deshabilitadas.
- El CSS asociado usa fuente Sono desde Google Fonts para todos los pseudo-selects.
- Se usa en Cliente y Comprobante que modifica dentro de Comprobante Venta.

### `numero_argentino.js`

Utilidad global para parsear/formatear numeros con convencion argentina. Debe usarse para importes, cantidades y bonificaciones visibles. Regla de UI: el punto del teclado numerico se interpreta como coma decimal donde aplica.

### `fecha_argentina_datepicker.js`

Utilidad global para inputs de fecha. Contrato visual: fechas visibles en `DD/MM/YYYY`; entradas internas o normalizadas pueden validarse hacia ISO cuando el backend lo requiera.

# 3. Contratos funcionales criticos

## 3.1 Ventas y comprobantes

- Tipos funcionales vigentes: FACTURA, NOTA_DEBITO y NOTA_CREDITO.

- Codigos fiscales clase C usados en la UI actual: 011 Factura C, 012 Nota de Debito C, 013 Nota de Credito C.

- Numero visible: FC/ND/NC C 0001-00000001.

- Proximo numero se calcula por tipo_comprobante + letra + punto_venta.

- ND debe asociarse a una FC confirmada del mismo cliente antes de confirmar. NC debe asociarse a una FC o ND confirmada del mismo cliente, con saldo disponible para NC.

- La asociacion queda registrada en ventas_comprobantes_asociaciones con tipo_relacion MODIFICA.

- No hay CAE, vencimiento CAE ni autorizacion ARCA implementada en esta etapa.

### 3.1.1 Actualizacion complementaria del alta de comprobante de venta

Contrato UI vigente del formulario `Gestion/Ventas/Comprobantes/Nuevo`:

- Fila superior: Tipo, Letra, Punto venta, Numero y Moneda.
- Cliente queda debajo y usa pseudo-select con busqueda.
- Fecha default: fecha actual del servidor al renderizar contexto.
- Vencimiento default: fecha + 30 dias.
- Si el usuario modifica Fecha en pantalla, Vencimiento se actualiza como Fecha + 30 dias.
- Al cambiar Tipo, se actualizan Letra, Punto venta, Numero y Moneda segun metadata del tipo seleccionado.
- El numero mostrado es orientativo/UX; el service recalcula el numero definitivo al crear el comprobante.

Contrato de dominio agregado:

- Si `fecha_vencimiento` llega vacia al service, se guarda como `fecha + 30 dias`.
- Si `fecha_vencimiento` llega informada, se respeta.
- La regla de vencimiento no depende solamente de JavaScript.


## 3.2 Impacto contable de ventas

- Confirmar un comprobante de venta genera asiento contable automatico confirmado tipo VENTA.

- Descripcion de cabecera y lineas de asiento: Comprobante: FC/ND/NC C 0001-00000001 | Sujeto: Cliente.

- FC y ND imputan cuenta deudores por ventas al DEBE y cuenta de ingreso al HABER.

- NC invierte el asiento: deudores al HABER e ingreso/resultado al DEBE.

- El asiento queda vinculado a ventas_comprobantes.asiento_id.

## 3.3 Cuenta corriente cliente

- Confirmar FC crea movimiento FACTURA al DEBE.

- Confirmar ND crea movimiento NOTA_DEBITO al DEBE.

- Confirmar NC crea movimiento NOTA_CREDITO al HABER.

- Descripcion persistida para movimientos de venta nuevos: solo el comprobante, ejemplo FC C 0001-00000001.

- Pantalla de mayor: Detalle muestra solo FC/ND/NC C punto-numero; Movimiento queda oculto.

- El saldo se calcula en lectura; no se persiste saldo acumulado por movimiento como dato fuente.

# 7. Estado de calidad y trazabilidad

- Migraciones ordenadas y validadas hasta 026.

- Servicios mantienen SQL fuera de rutas; repositorios concentran persistencia SQLite.

- Montos monetarios principales se manejan como INTEGER en centavos; cantidades y cotizaciones usan escalas enteras especificas.

- Ultimos contratos incorporados: asientos de ventas tipo VENTA, descripcion contable Comprobante+Sujeto, cuenta corriente cliente con detalle solo comprobante y columna Movimiento oculta.

- Este documento no aplica cambios de codigo ni de base de datos; solo consolida el estado funcional y estructural vigente.
